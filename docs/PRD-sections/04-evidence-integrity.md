### 법적 근거 요약

대한민국 법원이 인정하는 디지털 증거의 3대 요건:

1. **동일성**: 원본과 사본이 동일함을 증명 (해시값 비교)
2. **무결성**: 수집 시점부터 법정 제출까지 변경되지 않음을 보증
3. **신뢰성**: 수집 과정에서 적용된 기술의 신뢰성 및 타당성 입증

관련 규정:
- 대검찰청 예규 "디지털 증거의 수집·분석 및 관리 규정"
- 법원 행정규칙 "디지털 증거의 처리 등에 관한 규칙"
- 형사소송법 제313조 (진술서 등의 증거능력), 제292조의2 (디지털 증거)

## 3.5.1 해시 생성 및 검증

| 항목 | 내용 |
|------|------|
| **ID** | FR-EV-001 |
| **제목** | 채증 파일 SHA-256 해시값 자동 생성 |
| **설명** | 모든 채증 파일(스크린샷, HTML, WARC, 메타데이터 등) 생성 즉시 SHA-256 해시값을 자동으로 계산하여 기록한다. 해시 생성은 파일 저장과 원자적(atomic)으로 수행되어야 하며, 해시 생성 실패 시 채증 작업을 실패로 처리한다. |
| **우선순위** | P0 |
| **Phase** | Phase 1 (MVP) |
| **수용 기준** | 1. 채증 완료 후 모든 파일에 대해 SHA-256 해시값이 생성된다 2. 해시값은 `metadata.json`에 파일명-해시값 쌍으로 기록된다 3. 동일 파일에 대해 해시값을 재계산하면 항상 동일한 결과를 반환한다 4. 해시 생성 실패 시 채증 상태가 `HASH_ERROR`로 변경된다 5. 해시 알고리즘은 SHA-256(FIPS 180-4)을 사용한다 |
| **기술 구현 참고** | Node.js `crypto` 모듈 (`crypto.createHash('sha256')`) |

| **기술 구현 참고** | Node.js `crypto` 모듈, `fs.createReadStream` 스트리밍 해시 계산 |

| **ID** | FR-EV-004 |
| **제목** | OpenTimestamps (비트코인 블록체인) 타임스탬프 생성 |
| **설명** | 증거 패키지의 해시 매니페스트에 대해 OpenTimestamps를 사용하여 비트코인 블록체인 기반 타임스탬프를 생성한다. 생성된 `.ots` 파일은 증거 패키지에 포함된다. OpenTimestamps는 무료이며 탈중앙화 방식으로 10년 이상 검증 가능한 영속성을 보장한다. |
| **우선순위** | P0 |
| **Phase** | Phase 1 (MVP) |
| **수용 기준** | 1. 증거 패키지 생성 시 `timestamp_proof.ots` 파일이 자동 생성된다 2. `.ots` 파일이 `hash_manifest.sha256`의 해시를 포함한다 3. `ots verify` 명령으로 타임스탬프 검증이 성공한다 4. 비트코인 블록 확인(confirmation)까지 평균 1~2시간 소요되며, 확인 상태를 추적한다 5. OpenTimestamps 캘린더 서버 연결 실패 시 재시도(3회)하고, 최종 실패 시 경고 로그를 기록하되 채증은 계속 진행한다 |
| **기술 구현 참고** | `opentimestamps` CLI 또는 JavaScript 클라이언트, 공개 캘린더 서버(a.pool.opentimestamps.org 등) |
| **의사결정 참조** | `docs/planning/decisions.md` — "[2026-03-14] 증거 무결성: OpenTimestamps 즉시 도입" (확정) |

| **기술 구현 참고** | `pdf-rfc3161` 라이브러리 참고하여 경량 HTTP TSA 클라이언트 구현. TSA 서버: FreeTSA.org (무료), DigiCert (유료 백업) |

## 4.2 보안 요구사항

| 항목 | 내용 |
|------|------|
| **ID** | NFR-SEC-001 |
| **제목** | 증거 파일 암호화 저장 |
| **설명** | 저장소에 보관되는 모든 증거 파일을 암호화하여 비인가 접근 시에도 내용 열람이 불가능하도록 한다. 전송 중(in-transit) 및 저장 시(at-rest) 암호화를 모두 적용한다. |
| **우선순위** | P1 |
| **관련 규정** | 개인정보보호법 제29조(안전조치 의무), 개인정보의 안전성 확보조치 기준(고시) |
| **수용 기준** | 1. 저장 시 암호화: AES-256-GCM 이상의 암호화가 적용된다 2. 전송 중 암호화: TLS 1.3이 적용된다 3. 암호화 키는 증거 파일과 분리된 키 관리 시스템(KMS)에서 관리된다 4. 암호화/복호화 이벤트가 감사 로그에 기록된다 5. 키 로테이션 정책이 수립되어 있다 (최소 연 1회) |
| **기술 구현 참고** | MinIO Server-Side Encryption (SSE-S3), Node.js `crypto.createCipheriv('aes-256-gcm')` |

| **ID** | NFR-SEC-002 |
| **제목** | 역할 기반 접근 권한 관리 (RBAC) |
| **설명** | 시스템 사용자별 역할에 따라 증거에 대한 접근 권한을 제어한다. 최소 권한 원칙(Principle of Least Privilege)을 적용하여 각 역할에 필요한 최소한의 권한만 부여한다. |
| **우선순위** | P1 |
| **관련 규정** | 개인정보보호법 제29조, 정보보호 관리체계(ISMS) 기준 |
| **수용 기준** | 1. 최소 4개 역할이 정의된다: `ADMIN`(전체 관리), `INVESTIGATOR`(채증 실행/조회), `REVIEWER`(증거 조회/검증), `AUDITOR`(감사 로그 조회) 2. 각 역할에 CRUD 수준의 권한이 매핑된다 3. 증거 삭제 권한은 `ADMIN`에게만 부여되며, 삭제 시 사유 기록이 필수이다 4. 권한 변경 시 감사 로그에 기록된다 5. 세션 타임아웃이 설정된다 (기본 30분) 6. 동시 로그인 제한이 가능하다 |
| **기술 구현 참고** | NextAuth.js v5, PostgreSQL 역할/권한 테이블, 미들웨어 기반 접근 제어 |

| **제목** | 감사 로그 변조 방지 |
| **설명** | 감사 로그의 무결성을 보장하여 로그 자체가 증거로서의 신뢰성을 유지하도록 한다. 로그는 append-only 방식으로 기록되며, 변조 탐지 메커니즘이 적용된다. |
| **우선순위** | P1 |

| **기술 구현 참고** | PostgreSQL `REVOKE UPDATE, DELETE ON audit_log`, 해시 체인(각 레코드에 `prev_hash` 필드), 별도 읽기 전용 복제본 |
