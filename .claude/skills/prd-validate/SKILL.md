# PRD 기술적 검증

PRD를 Chain of Thought(단계별 추론) 방식으로 체계적으로 기술 검증합니다.

> **참고**: 정밀한 검증이 필요한 경우 opus 모델 사용을 권장합니다.

## 사용법

```
/prd-validate [PRD 파일 경로 (기본: docs/PRD.md)]
```

## 5단계 검증 프로세스

### Step 0: 공식 문서 확인 및 사실 검증
- WebFetch/Context7로 API 공식 문서 직접 확인
- GitHub 예제 코드, 샘플 프로젝트 검토
- 최신 버전 및 변경사항 확인
- 확인 불가 시 [UNCERTAIN] 태그 + "공식 문서 확인 필요" 명시

### Step 1: 초기 분석 및 가설 설정
- 프로젝트 유형, 기술 스택, 외부 API 의존성, 핵심 기능 파악
- 초기 가설 설정: "이 PRD는 __를 구현하려고 하며, 주요 도전은 __ 일 것이다"

### Step 2: API/라이브러리 기능 검증
- 각 기술적 주장을 개별 검증
- 이 스킬 디렉토리의 `tagging-guide.md` 참조하여 태깅

### Step 2.5: 대안 탐색
- "구현 불가능" 판단 전 3개 이상 대안 검토 필수
- 직접적 대안, 우회적 해결, 단계적 구현, 아키텍처 조정

### Step 3: 논리적 일관성 검증
- 데이터 플로우 추론, 기능 간 호환성
- 사용자 여정 vs 기술 구현 일치성

### Step 4: 복잡도 및 위험도 평가
- 1인 개발자 관점 구현 복잡도 (1-5점)
- 기본 기능, API 통합, 보안 구현 각각 평가

### Step 5: 가설 검증 및 수정
- 초기 가설 vs 검증 결과 비교
- 수정된 종합 결론 도출

## 환각 방지 원칙

- API 기능을 추측하지 않음
- 라이브러리 기능을 가정하지 않음
- "구현 불가능" 성급히 판단하지 않음
- 부정적 편향을 피함

## 출력 형식

이 스킬 디렉토리의 `report-template.md`를 따라 검증 결과를 출력합니다.

### 최종 판정 (5단계)

- **검증 완료**: PRD 그대로 구현 가능
- **조건부 통과**: 수정 후 구현 가능
- **대규모 수정 필요**: 아키텍처 재설계 필요
- **부분 구현 가능**: 범위 축소 필요
- **재검토 필요**: 전면 재작성 필요

### 신뢰도 지표

- 기술적 신뢰도: __/10
- 구현 복잡도: __/10
- 외부 의존 위험: __/10
- 전체 위험도: __/10

## 규칙

- 모든 검증 내용은 **한국어**로 작성
- 모든 진술에 태깅 필수 (`tagging-guide.md` 참조)
- 확인 없이 사실로 제시 금지
